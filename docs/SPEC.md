# ТЗ: Універсальний парсер графіків відключень електроенергії

> Статус: чернетка на узгодження. Рекомендовані рішення позначені як **(рекомендовано)** — будь-яке можна змінити перед стартом розробки.

## 1. Контекст і мета

Потрібен сервіс, який автоматично (через GitHub Actions, кожні ~5 хв) збирає актуальні
графіки відключень електроенергії з сайтів операторів розподілу та публікує їх у вигляді
зручного для людини й машини JSON. Архітектура — **універсальна**: одне ядро + набір
**адаптерів на кожне джерело/регіон**. Перші два джерела:

- `https://www.dtek-krem.com.ua/ua/shutdowns` — ДТЕК Київські **регіональні** електромережі (Київська обл.).
- `https://www.dtek-kem.com.ua/ua/shutdowns` — ДТЕК Київські електромережі (м. Київ).

Дані потрібні **по кожній групі та підгрупі** черг відключень. Надалі плануються інші джерела
та окреме репо з інтеграцією для Home Assistant (поза межами цієї задачі, але формат даних
проєктуємо з прицілом на машинне споживання).

Рішення має бути **самостійним** — не копіювати нейминг чи фрагменти коду сторонніх проєктів
(зокрема `Baskerville42/outage-data-ua`), хоча їхні підходи вивчені як орієнтир.

## 2. Результати розвідки сайтів (факти)

1. **WAF: Imperva Incapsula.** Голий HTTP-запит повертає ~212-байтний stub:
   `<script src="/_Incapsula_Resource?SWJIYLWA=...">` + cookies `visid_incap_*`, `incap_ses_*`.
   Реальні дані з'являються лише після виконання JS-челенджу в браузері → **обов'язково
   Playwright/Puppeteer**, не `fetch`/`curl`.
2. **Дані в DOM/JS.** Розклад вбудовано у глобальний об'єкт `window.DisconSchedule`:
   - `DisconSchedule.preset` — тижневий **шаблон**: `preset[group][dow][hour]`,
     де `group` = код групи/підгрупи (`"1.1"`, `"2.2"`…), `dow` = ISO день тижня `"1".."7"`,
     `hour` = `"1".."24"`;
   - `DisconSchedule.fact` — **фактичні** вікна на сьогодні/завтра:
     `fact[timestamp].day_data[group][hour]` (той самий формат годин).
   Маркер «сторінка завантажилась успішно» = `preset` непорожній.
3. **Кодування години** (значення слота, підтверджено по референсних реалізаціях):

   | код | стан | у нашій схемі |
   |---|---|---|
   | `yes` | світло є | (не генерує інтервал) |
   | `no` | відключення цілу годину | `kind: off` |
   | `maybe` | можливе відключення цілу годину | `kind: possible` |
   | `first` / `second` | відключення в 1-шу / 2-гу півгодину | `kind: off` (півінтервал) |
   | `mfirst` / `msecond` | можливе відключення в 1-шу / 2-гу півгодину | `kind: possible` (півінтервал) |

   **Тип події** (`type`): з `preset` → `planned` (для `no`) / `possible` (для `maybe`);
   з `fact` → `emergency` / `stabilization` / `planned` (якщо запис fact несе тип; інакше
   маппимо як preset). `fact` **накладається поверх** `preset` на свою дату.
   > ⚠️ Точну форму поля типу всередині `fact[timestamp]` (де саме лежить emergency/stabilization)
   > звіримо на першому живому запуску; `raw`-снапшот гарантує, що нічого не втратимо.
4. **Затримка під навантаженням.** Сайт може показувати повідомлення «зачекайте кілька хвилин,
   дані завантажаться автоматично». Парсер має це враховувати: чекати появи `DisconSchedule`
   з таймаутом і ретраями, а не падати одразу.
5. Обидва сайти **ідентичні** за структурою → спільний DTEK-адаптер, параметризований
   доменом/регіоном.

## 3. Рішення (рекомендовані за замовчуванням)

| Питання | Рішення | Чому |
|---|---|---|
| Рантайм | **Node.js (чистий JS, ESM)** ✅ узгоджено | Без кроку компіляції; контракт схеми документуємо через JSDoc + опційний JSON Schema |
| Браузер | **Playwright + Chromium** (рекомендовано) | Краще API контекстів/cookies/stealth, ніж Puppeteer; активна підтримка |
| Публікація | **Окрема orphan-гілка `data`** ✅ узгоджено | Код у `main`, дані в `data` — чиста історія, легко споживати через `raw.githubusercontent.com` |
| Формат | **Нормалізована схема + сирий снапшот** ✅ узгоджено | Стабільний контракт для споживачів + страховка від змін на сайті/для дебагу |
| Файли | **Файл на джерело + зведений `index.json`** ✅ узгоджено | Баланс зручності для людини й машини |

## 4. Архітектура

```
src/
  core/
    schema.js         # константи версії схеми + JSDoc-typedef доменних об'єктів
    browser.js        # запуск Playwright, stealth, проходження Incapsula, ретраї
    pipeline.js       # orchestrate: для кожного джерела → fetch → parse → normalize → write
    time.js           # робота з Europe/Kyiv, ISO 8601, злиття суміжних слотів у інтервали
    publish.js        # запис у вихідну теку + хеш контенту + статус оновлення
    logger.js
  sources/
    registry.js       # реєстр джерел (id, назва, регіон, url, adapter)
    dtek/
      adapter.js      # спільна логіка DTEK: дістати DisconSchedule, розпарсити preset/fact
      dtek-krem.js    # конфіг джерела: Київська обл.
      dtek-kem.js     # конфіг джерела: м. Київ
  index.js            # CLI-точка входу (всі джерела / одне за --source)
data/                 # (у гілці data) вихідні JSON
  index.json
  dtek-krem.json
  dtek-kem.json
schemas/              # (опційно) JSON Schema для валідації вихідних файлів
docs/SPEC.md
.github/workflows/parse.yml
package.json          # "type": "module"
```

**Контракт адаптера** (щоб додавати джерела однаково; документуємо через JSDoc):
```js
/**
 * @typedef {Object} SourceAdapter
 * @property {string} id            // "dtek-krem"
 * @property {string} displayName   // "ДТЕК Київські регіональні електромережі"
 * @property {string} region        // "Київська область"
 * @property {string} url
 * @property {(browser) => Promise<RawSnapshot>} fetch    // сирий DisconSchedule + html-метадані
 * @property {(raw: RawSnapshot) => NormalizedSchedule} parse  // → нормалізована схема
 */
```

## 5. Схема вихідного JSON (нормалізована)

`data/<source>.json`:
```jsonc
{
  "schemaVersion": "1.0",
  "source": {
    "id": "dtek-krem",
    "name": "ДТЕК Київські регіональні електромережі",
    "region": "Київська область",
    "url": "https://www.dtek-krem.com.ua/ua/shutdowns"
  },
  "updatedAt": "2026-06-29T12:00:00+03:00",     // час успішного парсингу (Europe/Kyiv, ISO 8601)
  "status": {                                    // результат останнього запуску
    "ok": true,
    "code": "ok",                                // ok | waf_blocked | timeout | parse_error | no_data
    "message": null,
    "contentHash": "sha256:…"                    // для виявлення фактичних змін
  },
  "groups": ["1.1", "1.2", "2.1", "…"],          // усі знайдені групи/підгрупи (відсортовані)
  "schedules": {                                  // НОРМАЛІЗОВАНО — головне для споживача
    "1.1": {
      "group": "1",
      "subgroup": "1",
      "intervals": [                              // плаский список датованих інтервалів
        { "start": "2026-06-29T08:00:00+03:00",
          "end":   "2026-06-29T12:00:00+03:00",
          "kind":  "off",                         // off | possible
          "type":  "planned",                     // planned | possible | emergency | stabilization
          "origin": "preset" }                    // preset | fact (звідки взято)
      ]
    }
  },
  "raw": { "preset": { /* як на сайті */ }, "fact": { /* як на сайті */ } }
}
```

`data/index.json` — зведення по всіх джерелах: список `{id, name, region, updatedAt, status, file}`
+ глобальний `generatedAt`. Зручно для дашборду/HA — один запит, щоб дізнатись що оновлено.

**Принципи нормалізації:**
- Час — **Europe/Kyiv**, ISO 8601 з офсетом. `preset` (тижневий шаблон) **розгортається на
  горизонт** ~сьогодні..+6 днів у конкретні дати; `fact` **накладається поверх** preset на свої дати.
- Слот→інтервал за таблицею кодування (розділ 2.3); `yes` інтервалів не дає (показуємо лише відключення).
- Півгодинні коди (`first/second/mfirst/msecond`) → інтервали по 30 хв.
- Суміжні однакові інтервали (той самий `kind`+`type`) **зливаємо** в один.
- При помилці парсингу/WAF **зберігаємо попередні `groups/schedules/raw`**, оновлюємо лише
  `status` + `updatedAt` (не публікуємо порожнечу).

## 6. Проходження Incapsula та стійкість (core/browser.js)

- Реальний Chromium, контекст із `locale: uk-UA`, `timezoneId: Europe/Kyiv`, реалістичний User-Agent.
- **Персистентні cookies між запусками** (кеш у артефакті/гілці) — менше челенджів.
- Stealth-патчі: `navigator.webdriver`, `languages`, `plugins`, `--disable-blink-features=AutomationControlled`.
- Очікування `DisconSchedule` через `page.waitForFunction` з таймаутом (напр. до 3–5 хв сумарно,
  щоб покрити повідомлення «зачекайте кілька хвилин»).
- **Ретраї з експоненційним backoff** (напр. 3 спроби) + детект WAF-stub (короткий HTML без
  `DisconSchedule`) → код `waf_blocked`.
- Якщо всі спроби невдалі — не валимо весь пайплайн, фіксуємо статус для цього джерела, інші продовжуємо.

## 7. GitHub Actions (.github/workflows/parse.yml)

- Тригери: `schedule: cron */5 * * * *` + `workflow_dispatch`.
- Кроки: checkout → setup-node → `npm ci` → `npx playwright install --with-deps chromium` (з кешем)
  → `npm run parse` → publish у гілку `data`.
- Публікація в `data`: checkout гілки `data` у підпапку (або `git worktree`), перезапис JSON,
  коміт **тільки якщо `contentHash` змінився**, push. Уникаємо порожніх комітів щоп'ять хвилин.
- Concurrency-guard (щоб запуски не накладались), розумні таймаути job.
- **Токен:** PAT не потрібен — пуш у гілку `data` того ж репо робиться вбудованим
  `GITHUB_TOKEN` з `permissions: contents: write`. PAT знадобиться лише якщо колись
  захочемо тригерити інший workflow цим пушем або писати в інший репозиторій.
- ⚠️ GitHub cron не гарантує точність 5 хв під навантаженням, і `schedule`-workflow має лежати
  в **default-гілці** (активується після мержу `main`) — закладаємо це в очікування.

## 8. Етапи розробки

1. **Каркас**: package.json (`type: module`), lint/format, `core/schema.js` (JSDoc-типи), реєстр джерел.
2. **browser.js**: запуск Playwright + проходження Incapsula + ретраї (перевірити на реальному сайті).
3. **DTEK-адаптер**: дістати `DisconSchedule`, **зафіксувати реальну форму preset/fact**, парсер.
4. **Нормалізація + publish** (схема, злиття інтервалів, хеш, index.json).
5. **Друге джерело** `dtek-kem` (переважно конфіг) — валідація універсальності.
6. **GitHub Actions** workflow + гілка `data` + перший живий запуск.
7. README з документацією схеми та інструкцією додавання нового джерела.

## 9. Перевірка (acceptance)

- Локально: `npm run parse -- --source dtek-krem` створює валідний `data/dtek-krem.json` з
  непорожніми `groups` (усі групи/підгрупи) і коректними ISO-часами в Europe/Kyiv.
- Те саме для `dtek-kem`; `index.json` містить обидва джерела.
- Симуляція WAF/таймауту → статус `waf_blocked`/`timeout`, попередні дані збережені.
- Workflow у Actions проходить «зелено» і пушить у `data` лише за реальної зміни даних.
- JSON валідується проти схеми (за бажання — додати JSON Schema у `schemas/`).

## 10. Узгоджені рішення та залишкові питання

Узгоджено з замовником:
1. Рантайм — **Node.js, чистий JS (ESM)**.
2. Публікація — **orphan-гілка `data`**.
3. Формат — **нормалізована схема + сирий снапшот**, **файл на регіон** + `index.json`.

Залишкове (фіксуємо емпірично на кроці 3 розробки, не блокує старт):
- Точна granularity слотів DTEK (24 погодинних vs 48 півгодинних) та точні коди станів
  у `DisconSchedule` — визначимо на першому реальному запуску Playwright.
